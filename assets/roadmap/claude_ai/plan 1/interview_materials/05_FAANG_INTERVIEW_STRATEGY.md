## üí° Interview Tips from Successful Candidates

### Do's ‚úÖ

```
1. Think Out Loud
   "I'm thinking of using a HashMap because..."
   Interviewers can't read your mind!

2. Ask Clarifying Questions
   "Can I assume the input is sorted?"
   "What's the expected size of the data?"
   Shows you think about edge cases

3. Start with Brute Force
   "The naive solution would be O(n¬≤), but we can optimize..."
   Shows problem-solving process

4. Explain Trade-offs
   "We could use BLoC for better testability, but Provider
    is simpler and sufficient for this use case"

5. Test Your Code
   Walk through with example inputs
   Consider edge cases

6. Be Honest
   "I don't know X, but here's how I'd approach learning it"
   Better than making things up

7. Show Enthusiasm
   Companies want people who WANT to work there
```

### Don'ts ‚ùå

```
1. Don't Stay Silent
   Interviewer has no idea what you're thinking

2. Don't Jump to Code
   Understand the problem first
   Discuss approach before coding

3. Don't Ignore Hints
   Interviewers give hints to help you!
   "Have you considered using a HashMap?"

4. Don't Say "That's Easy"
   Even if it is. Stay humble.

5. Don't Bad-mouth Previous Companies
   Shows unprofessionalism

6. Don't Give Up
   "I'm stuck" ‚Üí "Let me think about this differently..."

7. Don't Neglect Behavioral Prep
   Technical skills aren't enough
```

## üé≠ Behavioral Interview Prep

### STAR Method Template

```
Situation: (20% of answer)
- Set the context
- Keep it brief

Task: (20% of answer)
- What needed to be done
- What was the challenge

Action: (40% of answer)
- What YOU specifically did
- How you approached it
- Why you made those choices

Result: (20% of answer)
- What was the outcome
- Metrics if possible
- What you learned
```

### Example Stories to Prepare (Have 10+ ready)

```
1. Biggest technical achievement
2. Most challenging bug you fixed
3. Time you disagreed with your manager
4. Time you had to make a trade-off decision
5. Time you failed and learned from it
6. Time you helped a teammate
7. Time you improved a process
8. Time you dealt with ambiguity
9. Time you had to learn something quickly
10. Time you led a project
```

### Example Answer (Meta E5 Interview)

**Question:** "Tell me about a time you had to make a technical trade-off"

**Answer:**

```
Situation:
I was leading the development of our Flutter app which needed
offline support. We had 3 options: SQLite, Hive, or Isar.

Task:
Choose the right database that would:
- Support complex queries (we had 10+ tables)
- Perform well (app already had performance issues)
- Be maintainable by our team

Action:
I created a spike for each option:
1. Benchmarked performance with our actual data
2. Assessed learning curve for team
3. Evaluated migration complexity

Results:
- SQLite: Slower queries but team knew SQL
- Hive: Fastest but limited query capabilities
- Isar: Best performance + query support but newest

Decision: Isar
- 3x faster than SQLite in our tests
- Full query support unlike Hive
- Risk: Newer library, but community was active

Result:
- App queries 3x faster
- Reduced offline sync time by 60%
- Team learned Isar in 1 week
- No issues in 6 months of production

Learning:
- Data-driven decisions beat assumptions
- Consider team capabilities in tech choices
- Sometimes newer tech is worth the risk
```
